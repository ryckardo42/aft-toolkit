🔄 Sync DET — arquitetura completa

Extensão v2.0.0 para Chrome e Edge (Manifest V3) · página na Chrome Web Store
Como a extensão captura o token do DET, quem fala com a API oficial e como as notificações chegam ao memory.md de cada OS.

A ideia em uma frase: a extensão empresta o crachá (o token de sessão que o próprio site do DET usa) e entrega a um destino de confiança — o painel local do AFT Toolkit (na sua máquina) ou o SisOS (na nuvem) — que usa esse crachá para consultar a API oficial do DET e gravar as notificações na ficha certa. A extensão nunca lê a tela, nunca vê os dados das notificações.

3. Os 4 moradores da extensão

manifest.json — a certidão de nascimento

Declara o que a extensão é e o que pode fazer. Pontos-chave:

  • Permissões: storage (guardar token e configuração), tabs (abrir/focar a aba do DET), webRequest (observar os cabeçalhos das requisições — é assim que o token é capturado).
  • Hosts autorizados: só o DET, o SisOS e 127.0.0.1/localhost. Nenhum outro site é visível para a extensão.
  • Registra o background.js como service worker, injeta o content.js apenas no domínio do DET e define o popup.html como a janelinha do ícone.
background.js — o vigia (service worker)

Roda em segundo plano, sem interface. Duas funções:

  • Capturar o token: via chrome.webRequest.onBeforeSendHeaders, observa as requisições que o próprio site do DET faz e extrai o cabeçalho Authorization: Bearer <JWT>. Valida o formato (3 partes; payload com exp/sub/cif/unique_name), ignora duplicatas, grava em chrome.storage.local com timestamp e avisa a aba do DET.
  • Proxy do sync local: quando o content.js pede, é ele quem faz o POST http://127.0.0.1:8347/api/det-sync — no service worker, com host_permission, não existe bloqueio de CORS.
Por ser MV3, o service worker "dorme" quando ocioso e acorda nos eventos. Por isso o token também fica no storage.local — sobrevive ao cochilo e a recarregamentos da aba (mas expira em ~25 min).
content.js — o rosto na página do DET
  • Injetado somente em auditor-det.sit.trabalho.gov.br.
  • Desenha o botão flutuante "Sincronizar" (canto inferior direito) e os toasts de resultado.
  • No clique: lê a configuração (storage.sync), pega o token (memória, com fallback no storage.local se tiver menos de 25 min) e dispara para os destinos configurados — SisOS via fetch direto, painel local via mensagem ao background.
  • Mostra o resultado: ✅ Local: 2 importada(s) · 1 prazo(s) atualizado(s) — incluindo, quando houver, ↳ N de outra fiscalização (ignoradas).
popup.html + popup.js + popup.css — o painel de controle
  • Abre ao clicar no ícone da extensão. Campos: URL/token do SisOS (opcionais) e a caixa PAINEL LOCAL (AFT TOOLKIT) com a URL (padrão http://127.0.0.1:8347).
  • Salva em chrome.storage.sync — a configuração acompanha o perfil do Chrome/Edge entre máquinas.
  • Tem um indicador de status do token e um botão "Abrir DET".
Corrigido em 22/07/2026: o indicador lia storage.session['det_access_token'], mas o token é gravado em storage.local['sisos_det_token'] — mostrava "Token não encontrado" mesmo com o sync funcionando. Agora lê o lugar certo e aplica a mesma validade de 25 minutos que o clique do botão usa, então o indicador diz exatamente o que o sync fará.
O arquivo injected.js na pasta do fonte é um resto da v1 — não é referenciado pelo manifest.json e não é executado nem empacotado.

4. O fluxo completo, passo a passo

  1. Você navega logado no DET. O site Angular faz suas próprias chamadas à API, cada uma levando o token de sessão no cabeçalho.
  2. background.js captura o token desses cabeçalhos (nunca da tela, nunca da sua senha), valida e guarda com timestamp.
  3. Você clica em "Sincronizar" (botão do content.js na página do DET).
  4. O token viaja para o painel local: content.js → background.js → POST http://127.0.0.1:8347/api/det-sync com corpo {"det_access_token": "..."}. Tudo dentro da sua máquina.
  5. servir_painel.py recebe e chama det_sync.py, que percorre cada pasta de OS ATIVAS/ que tenha memory.md.
  6. Para cada OS, det_sync.py consulta a API do DET (por CNPJ/CPF; RI como fallback), usando o token emprestado — a mesma consulta que a tela "Notificações" faz.
  7. Filtros: só notificações Confirmadas (status=1) com data de lavratura, e só do(s) RI(s) declarado(s) no campo ri: do front-matter — o identificador canônico da auditoria (ver seção 6). As de outras fiscalizações voltam como "ignoradas" no relatório — nunca em silêncio.
  8. Escrita no memory.md (com backup antes): notificação nova vira - [ ] CÓDIGO — prazo dd/mm/aaaa + uma sub-linha de detalhes (lavrada · ciência · última entrega · Confirmada [· ⚠️ atualização pendente]); prazo que mudou é atualizado na linha existente; o checkbox [ ]/[x] nunca é alterado — respondida é decisão do AFT. O alerta "atualização pendente" é dispensável no painel (um clique grava <!-- visto: -->) e só volta se a empresa fizer entrega nova.
  9. O resultado volta pelo mesmo caminho e aparece como toast na página do DET. O painel (painel.html) passa a mostrar os novos vencimentos.

5. A API do DET em detalhe

Endpoint (o mesmo que alimenta a tela "Notificações" do site):

POST https://auditor-det.sit.trabalho.gov.br/services/auditor/v1/notificacoes/pesquisa
Authorization: Bearer <token de sessão do AFT>
Content-Type: application/json

Corpo da pesquisa enviado pelo det_sync.py:

{
  "isPesquisaPadrao": false,
  "niEmpregador": "<CNPJ/CPF ou null>",   ← campo principal
  "ri": "<RI, só se não houver CNPJ>",
  "isSomenteMinhas": true,                 ← só notificações do próprio AFT
  "sequencia": 0, "ordenacaoCampo": "id", "ordenacaoDesc": false, ...
}

Constatado em produção (19–22/07/2026): a API pode continuar devolvendo itemAtualizado=true mesmo depois de o triângulo sumir da tela do DET (a tela apaga quando o AFT abre a notificação; o campo, não necessariamente). Por isso o alerta local é dispensável: um clique no painel grava <!-- visto: última entrega --> na sub-linha, e o alerta só reaparece se itemDataUltimaEntrega mudar (entrega nova = novidade real).

6. O filtro de RI — por que existe e como funciona

A pesquisa é por empregador, então o DET devolve notificações de todas as fiscalizações já feitas naquele CNPJ — inclusive antigas, de anos anteriores. O que amarra uma notificação à auditoria atual é o RI.

RIs "desta OS" = somente o(s) declarado(s) no campo ri: do front-matter do memory.md — o RI é o identificador canônico da auditoria. Para uma OS que acompanha duas fiscalizações do mesmo empregador (ex.: ação fiscal + investigação de acidente), declare os dois no próprio campo, separados por vírgula: ri: "320038432, 320199999". Nada é inferido de códigos já registrados na ficha. Se a ficha ainda não tem ri:, o sync adota o da notificação mais recente, grava no front-matter, e daí em diante vale a regra estrita.

Regra de ouro: notificação de RI alheio nunca é importada — e nunca é descartada em silêncio: volta no toast e no relatório como "ignorada", para o AFT saber que existe e decidir.

Histórico: em 19/07/2026 (caso MASTER AGROINDUSTRIAL) a regra foi restrita a códigos em linha de checkbox; em 22/07/2026 (caso CONSORCIO SQ, RI alheio 319969819) ela foi endurecida de vez — a união com RIs "da ficha" caiu, porque uma notificação antiga registrada um dia tornava o RI dela "conhecido" e puxava as irmãs para sempre. Agora o front-matter manda, sozinho.

7. O painel ativo (http://127.0.0.1:8347)

O destino local do sync é também um dashboard interativo das auditorias — 100% na sua máquina. O gerar_painel.py lê os memory.md de OS ATIVAS/ e monta um card por OS, colorido pela urgência dos prazos de DET; o servir_painel.py serve esse HTML e executa as ações mecânicas, sempre com backup prévio + edição cirúrgica da ficha. Aberto por duplo-clique no arquivo (file://), o painel fica somente leitura.

8. Segurança — o que a extensão nunca faz

  • Nunca vê sua senha — o login é no gov.br; a extensão só observa o token que o site já usa.
  • Nunca lê a tela do DET nem faz scraping — os dados vêm da API oficial, consultada pelo painel.
  • Nunca grava o token em arquivo — vive em memória e no storage do navegador, expira em ~25 min.
  • Nunca marca notificação como respondida nem apaga o que você escreveu — cada memory.md alterado recebe backup antes.
  • No modo painel local, nada sai para a internet além da consulta do próprio painel à API do governo — extensão ↔ painel é 100% 127.0.0.1.
  • Modo de inspeção: AFT_DET_DRYRUN=1 faz a consulta e devolve o relatório completo sem gravar nada.
